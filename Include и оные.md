# Правильный порядок `#include` в C++

## Канонический порядок (используй всегда)

```cpp
// 1. Собственный заголовок (если .cpp)
#include "my_class.h"

// 2. Стандартная библиотека C++
#include <algorithm>
#include <array>
#include <cmath>
#include <cstdint>
#include <iostream>
#include <vector>

// 3. Сторонние библиотеки
#include <QtCore/QObject>
#include <QtCore/QString>

// 4. Заголовки проекта
#include "utils/math.h"
#include "audio/wave.h"
```

---

## ЗАЧЕМ ТАКОЙ ПОРЯДОК

### 1️⃣ Свой заголовок — ПЕРВЫМ

```cpp
#include "my_class.h"
```

- ловит недостающие `#include` в `.h`
    
- гарантирует, что заголовок **самодостаточен**
    
- предотвращает “магические зависимости”
    

Если `.h` сломан — ты узнаешь сразу.

---

### 2️⃣ `<...>` стандартной библиотеки

```cpp
#include <vector>
#include <algorithm>
```

- стабильные
    
- независимые
    
- не должны зависеть от проекта
    

---

### 3️⃣ Сторонние библиотеки (Qt, Boost, SDL)

```cpp
#include <QtCore/QString>
```

- могут тащить макросы
    
- могут конфликтовать
    
- лучше держать отдельно
    

---

### 4️⃣ Заголовки проекта

```cpp
#include "utils/math.h"
```

- твой код
    
- зависит от всего выше

## `.h` vs `.cpp`

### В `.h`

- **минимум**
    
- только то, что нужно для объявления
    
- использовать forward declaration
    

```cpp
class Foo;     // вместо #include "foo.h"
```

---

### В `.cpp`

- можно всё
    
- здесь “тяжёлые” заголовки
    

---

## Частые паттерны

### Если файл — `.cpp`

```cpp
#include "this_file.h"
#include <vector>
#include <algorithm>
```

### Если файл — `.h`

```cpp
#pragma once
#include <cstdint>
#include <vector>
```

---

## Железное правило (запомнить)

> **Если удалил `#include` и код всё ещё компилируется — значит, он был лишний.**

---

## TL;DR

```
1. свой .h
2. <STL>
3. <3rd-party>
4. "project"
```

Этот порядок:

- ловит ошибки раньше
    
- масштабируется
    
- принят в крупных кодовых базах
